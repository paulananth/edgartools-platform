failed** (`States.ExceedToleratedFailureThreshold`, 93/108 succeeded, 2
aborted, 12 never started) — root cause fully diagnosed, no fix applied:

- The failing item's ECS task couldn't pull its image:
  `CannotPullContainerError: ... failed to resolve reference
  ...edgartools-prod-warehouse@sha256:aec65349d7d186c7109dac8f602d4dacd6b64a0f46c041ff71dca4669e5dc47b: not found`
- `edgartools-prod-warehouse`'s ECR lifecycle policy keeps only the 20
  most-recently-pushed images (any tag), with **no exception for images still
  referenced by an active task-definition revision**
  (`edgartools-prod-medium:56`, pinned by whatever registered that state
  machine). Given how many images get pushed per day (CI auto-push on every
  `main` commit, plus manual builds), an older pinned digest can age out from
  under a still-in-use task-def revision.
- This needs a real decision, not a quick patch: increase the retention
  count, exclude images referenced by in-use task-def revisions from the
  lifecycle policy, or move long-lived state machines to reference a mutable
  tag (`:prod`) instead of a specific digest. Flag this to the user for a
  decision before changing the lifecycle policy — it affects retention/cost
  tradeoffs across the whole registry, not just this one state machine.
- 93/108 of that batch's work already succeeded and is presumably intact;

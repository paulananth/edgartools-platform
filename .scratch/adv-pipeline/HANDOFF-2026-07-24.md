## Production incident that happened after the original handoff — READ THIS

While repairing the canonical `silver.duckdb` locally (see next section), a
diagnostic `aws s3api put-object` call was accidentally pointed at the **real
production key**
(`s3://edgartools-prod-warehouse-690839588395/warehouse/silver/sec/silver.duckdb`)
with a 22-byte throwaway test string, overwriting the real 884MB canonical.
This was caught immediately and **has already been fixed** — do not re-fix it,
just be aware of why the object's version history looks the way it does.

**Restored via a server-side `copy-object`** (no data transfer over any local
connection, so it's fast and 100% reliable) back to the last-known-good
pre-incident version:

```bash
aws s3api copy-object --region us-east-1 \
  --bucket edgartools-prod-warehouse-690839588395 \
  --key warehouse/silver/sec/silver.duckdb \
  --copy-source "edgartools-prod-warehouse-690839588395/warehouse/silver/sec/silver.duckdb?versionId=n5bUperzS5AwFCP_olcvvD_f79M8Ne.p"
```

Verified with real content checksums (not size/exit-code) — the restored
object's ETag/CRC32 match the pre-incident version exactly:

```
ETag:         "93b8595faf3afb1b2ca8a578c100dfd6"
ChecksumCRC32: Cwq5Vg==
ContentLength: 884748288
Live VersionId (as of this handoff): kiZ7eJFgcaHMpUyW97llH5o1Rs1HJzmI
```

**Important: this restored the STALE pre-migration version** — i.e. production
right now is back to a known-good state, but `sec_adv_private_fund.fund_index`
is still the old `SMALLINT`. The BIGINT-migrated repair (see below) still
needs to land. Before doing anything else, re-verify this is still the live
object (someone/something else may have touched it since):

```bash
aws s3api head-object --region us-east-1 \
  --bucket edgartools-prod-warehouse-690839588395 \
  --key warehouse/silver/sec/silver.duckdb --checksum-mode ENABLED
# Expect ETag "93b8595faf3afb1b2ca8a578c100dfd6" / ChecksumCRC32 Cwq5Vg==
# If VersionId is something other than kiZ7eJFgcaHMpUyW97llH5o1Rs1HJzmI and the
# checksum differs, something changed it after this handoff — investigate
# before assuming either "good" or "bad".
```

**Lesson recorded for both of us:** verify S3 uploads by comparing
`ETag`/checksum against a local ground truth (`shasum -a 256` locally, compare
to S3's returned checksum), never by `ContentLength`/`VersionId` alone — two
different files can be the same size, and a "successful" CLI exit code does
not guarantee the bytes actually landed under a flaky connection.

## Unrelated finding — do NOT fix as part of ADV work, but don't lose it

**Ticket 20's `ticket20-strict-q1y-20260724T004600Z` Distributed Map execution
failed** (`States.ExceedToleratedFailureThreshold`, 93/108 succeeded, 2
aborted, 12 never started) — root cause fully diagnosed, no fix applied:

- The failing item's ECS task couldn't pull its image:
  `CannotPullContainerError: ... failed to resolve reference
  ...edgartools-prod-warehouse@sha256:aec65349d7d186c7109dac8f602d4dacd6b64a0f46c041ff71dca4669e5dc47b: not found`
- `edgartools-prod-warehouse`'s ECR lifecycle policy keeps only the 20
  most-recently-pushed images (any tag), with **no exception for images still
  referenced by an active task-definition revision**
  (`edgartools-prod-medium:56`, pinned by whatever registered that state
  machine). Given how many images get pushed per day (CI auto-push on every
  `main` commit, plus manual builds), an older pinned digest can age out from
  under a still-in-use task-def revision.
- This needs a real decision, not a quick patch: increase the retention
  count, exclude images referenced by in-use task-def revisions from the
  lifecycle policy, or move long-lived state machines to reference a mutable
  tag (`:prod`) instead of a specific digest. Flag this to the user for a
  decision before changing the lifecycle policy — it affects retention/cost
  tradeoffs across the whole registry, not just this one state machine.
- 93/108 of that batch's work already succeeded and is presumably intact;
  the remaining 15 (2 aborted + 12 never-started) need a redrive once a
  task-def revision with a valid image is in place. This is a **separate**
  workstream from ADV — don't conflate fixing it with finishing ADV.

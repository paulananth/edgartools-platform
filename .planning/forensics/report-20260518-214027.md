## Problem Statement
`bootstrap-recent-10` (and earlier `bootstrap-phased`) have never completed since the
MDM+Neo4j pipeline stages were wired in on 2026-05-18. Every run has been manually
aborted before completion.

## Anomalies Found

### A1 — Silent Gold Publish (CONFIRMED ROOT CAUSE of perceived stuckness)
**Evidence:** `gold_publish_started` fired at 10:41:34; next log at 10:41:34 (`gold_build_started`
on rev 48+); then silence for 20+ minutes while `build_gold()` ran a Python loop calling
SHA-256 (`_det_key`) 2.7M times for `dim_filing` and `fact_filing_activity`.

**Impact:** Operators saw no logs → concluded stuck → killed the run.

**Resolution applied:**
- Rev 46: Added `gold_build_started/completed`, `gold_storage_write_started/completed`,
  `gold_snowflake_export_started/completed` events (obs commit `07a2c2b`)
- Rev 48: Added per-table `gold_table_started/completed` events (obs commit `5d67539`)
- Rev 49: Replaced 2.7M Python SHA-256 loops with DuckDB `hash()` in SQL —
  `dim_filing`: 27s → 2.4s, `fact_filing_activity`: 28s → 2.1s (perf commit `106014d`)

### A2 — `fact_ownership_transaction` Python Row Loop (ACTIVE — current run)
**Evidence:** `gold_table_started table=fact_ownership_transaction` at 01:37:56 with no
completion log yet. The builder uses `_fetch_rows()` + per-row Python `_det_key()` calls
on 34K non-derivative + 2K derivative ownership transactions joined to 2.7M filing rows.
This is the next remaining Python loop bottleneck after the dim_filing fix.

**Data:** `sec_ownership_non_derivative_txn`: 34,918 rows; `sec_ownership_derivative_txn`: 2,184 rows.
**Expected impact:** Lower than dim_filing (37K rows vs 2.7M) — likely 2-10 seconds, not stuck.

### A3 — New MDM/Neo4j Stages Never Validated (UNKNOWN RISK)
**Evidence:** The MDM chain (MdmRun → MdmBackfill → MdmSync → MdmVerify) was wired into
`bootstrap-recent-10` and `daily-incremental` on 2026-05-18 (commit `249b261`). Zero runs
have reached these stages. No evidence MDM credentials, connectivity, or silver source
configuration work correctly in this pipeline context.

**Risk:** MDM stages require `MDM_SILVER_DUCKDB` env var pointing to the S3 silver path,
MDM Postgres DSN, and Neo4j credentials — all injected via Secrets Manager. If any
credential or env var is misconfigured for these state machine stages, the pipeline will
fail at MdmRun, which comes AFTER the now-fast gold build.

### A4 — Repeated Kill-and-Restart Loop
**Evidence:** 6 executions in 24 hours, all aborted. Longest: 844.5 minutes (14 hours).
Commit history shows 4 observability/performance fixes shipped in one day.

**Pattern:** Each abort triggered a code fix → new image → new run → new abort. The
pipeline has never been allowed to run to natural completion or failure.

## Root Cause (5-Whys)

1. Pipeline never finishes → operator kills every run
2. Operator kills every run → gold publish appears silent/stuck
3. Gold publish appears silent → no progress logs for 20+ minutes
4. No progress logs → `build_gold()` loops 2.7M SHA-256 calls with no instrumentation
5. No instrumentation → gold_models.py was written as a batch computation with print-nothing-until-done design

**True root cause:** Gold build was an uninstrumented, 55-second Python loop with no
intermediate logging. Every run that reached gold publish was killed before it could complete.

---

## Recommendations

1. **Let the current run finish** — rev 49 gold build is fast (< 5s). The next blocker
   will be MDM stages, which have never been tested. Let it hit MdmRun naturally.

2. **Watch MDM connectivity** — If MdmRun fails, check: (a) `MDM_SILVER_DUCKDB` env var
   in the MDM task definition points to the correct S3 path; (b) Postgres DSN secret is
   current (deploy script refreshed it); (c) Neo4j secret is valid.

3. **Do not kill on silence** — Gold storage write (`write_gold_to_storage`) uploads 16
   Parquet files to S3. This can take 30-60 seconds with no per-file logging. Expected.

4. **Instrument `fact_ownership_transaction`** — Apply the same SQL rewrite used for
   `dim_filing` to push `_det_key` into DuckDB for the ownership fact rows.

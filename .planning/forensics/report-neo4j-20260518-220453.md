## Root Cause of "Not Loading" Perception

The pipeline had never successfully reached the MDM/Neo4j stages before today. Every
previous run was killed during the gold build phase due to a 17+ minute silent bottleneck
in `fact_ownership_transaction` (Python row loop on 37K rows). This made it appear that
MDM/Neo4j were broken — they were never given a chance to run.

**Timeline of fixes leading to first successful MDM entry:**

| Rev | Fix | Impact |
|-----|-----|--------|
| 46 | Added gold publish intermediate log events | Eliminated silent gap |
| 48 | Added per-table `gold_table_started/completed` | Identified bottleneck tables |
| 49 | `dim_filing` + `fact_filing_activity` SQL rewrite | 27s→2.4s each |
| 50 | `fact_ownership_transaction` + `fact_ownership_holding_snapshot` SQL rewrite | 17+min→<1s |

## Remaining Unknown: MdmSync (Neo4j write path)
`MdmSync` (`mdm sync-graph`) is the first stage that actually WRITES to Neo4j (upserts
nodes and edges). MdmRun only resolves MDM entities in Postgres. MdmSync has never run
and its connectivity/write behaviour is not yet validated. Monitor for failure at MdmSync.

## Recommendations
1. **Watch MdmSync** — the first actual Neo4j write attempt. Check logs for
   `ServiceUnavailable`, `AuthError`, or `ClientError` from the Neo4j driver.
2. **If MdmSync fails** — re-run `mdm check-connectivity --neo4j` to verify the Aura
   instance is still up, check for IP allowlist restrictions on the Aura console.
3. **Aura free-tier auto-pause** — Neo4j Aura free instances auto-pause after 72h of
   inactivity. The 2026-05-10 failure may have been an auto-pause event. Upgrade to
   a paid tier or ensure the instance is pinged periodically.

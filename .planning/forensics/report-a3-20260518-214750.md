## New Finding: fact_ownership_transaction â€” Secondary Gold Bottleneck

**Evidence:** `gold_table_started table=fact_ownership_transaction` at 01:37:56 UTC.
Still running 8.8+ minutes later with no `gold_table_completed` log.

**Root cause:** `_build_fact_ownership_transaction` calls `_ownership_fact_source_rows()`
which executes a UNION ALL JOIN between:
- `sec_ownership_non_derivative_txn` (34,918 rows) JOIN `sec_company_filing` (2,777,593 rows)
- `sec_ownership_derivative_txn` (2,184 rows) JOIN `sec_company_filing` (2,777,593 rows)

Possible causes of slowness:
1. **DuckDB JOIN on large table:** Even with hash join, materialising a 2.7M-row hash table
   takes time on a 2 GB task. After the SQL optimization of `dim_filing` (now returns
   a 2.7M-row PyArrow table held in memory) + `fact_filing_activity` (another 2.7M rows),
   the task may have < 1 GB free for this join.
2. **Memory pressure / disk spill:** DuckDB may be spilling to the 20 GB ECS ephemeral
   storage, making the JOIN 10â€“100Ă— slower.
3. **Python row materialisation:** After the JOIN runs, `_fetch_rows()` calls `fetchall()`
   which materialises 37K dicts in Python â€” this part is fast.

**Risk:** If memory pressure causes an OOM, ECS kills the container silently with no
log event. The Step Function will see `TaskFailed` with `StopCode: EssentialContainerExited`.

  <action>
    Create the `cagr(current_col, prior_col, years)` macro implementing
    `(current/prior)^(1/years) - 1` per D-02. Follow `safe_ratio_signed.sql`'s exact
    `case when <guards> then <formula> end` shape and whitespace byte-for-byte: 4-space
    indent for `case`, continuation `and` lines aligned with the extra-space indent.

    Guard requirements (D-02, GROW-02): the `then` branch fires ONLY when all four hold —
    `current_col is not null`, `prior_col is not null`, `current_col > 0`, `prior_col > 0`.
    Both operands STRICTLY POSITIVE — this is the broader D-02 decision, not a narrow
    same-sign check. Do NOT use `sign(current_col) = sign(prior_col)` or any same-sign
    logic: a negative-to-negative span (e.g. -100 -> -50) is mathematically computable but
    produces a misleadingly positive CAGR for a still-unprofitable company, which D-02
    prohibits (Pitfall 2). The strict-positive guard also keeps a negative base out of
    `power()`, whose negative-base/fractional-exponent behavior is undocumented in Snowflake
    (Pitfall 3) — defense-in-depth, do not weaken.

    Exponent (Pitfall 1, the single most likely silent-bug vector): render the exponent as
    float division, `power(... , 1.0 / {{ years }}) - 1`. Use the literal `1.0` numerator so
    the compiled SQL performs floating-point division. Integer division (`1 / {{ years }}`)
    truncates `1/3` and `1/5` to `0` in Snowflake, making `power(x, 0) = 1` and every CAGR
    silently `0`. Verify the COMPILED SQL text (next task's compile output) reads `1.0 / 3`
    and `1.0 / 5`, never `1 / 3`.

    Add a head comment explaining: FY-to-FY semantics, the strict-positive null guard
    rationale (D-02 + undocumented Snowflake negative-base behavior), and a warning not to
    weaken the guard. Describe the rationale by concept; do not paste any literal that a
    negative-grep gate keys on.
  </action>

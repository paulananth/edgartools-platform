<objective>
Create the `safe_ratio_signed` dbt macro — a sign-checked sibling of the existing
`safe_ratio` macro that nulls the ratio when the denominator is negative, zero, or null
(only computes when the denominator is strictly positive). This macro is the prerequisite
for `return_on_equity`'s D-01 negative-equity null guard, implemented in Plan 02.

Purpose: D-01 (CONTEXT.md) decided ROE must null when `total_equity < 0` per Damodaran's
treatment (negative/negative produces a misleadingly positive value). The model file's
100% convention is that every ratio factor calls a macro — never an inline `case` /
division. So the sign guard belongs in a reusable, centrally-documented macro, not inlined
in `financial_factors.sql`.
Output: One new macro file, `safe_ratio_signed.sql`.
</objective>

    (same parameter names and order as `safe_ratio`). The macro body is a single `case`
    expression with a `when` clause that requires: numerator is not null, denominator is
    not null, AND denominator `> 0`; the `then` branch is `numerator_col / denominator_col`;

    zero (already excluded by `safe_ratio`'s `<> 0`) and negative denominators in a single
    condition — do NOT add a separate `and denominator <> 0` clause (redundant) or a
    separate `case when denominator < 0 then null` branch (verbose). The ONLY business-logic
    difference from `safe_ratio.sql` is the comparison operator: `<> 0` becomes `> 0`. Match
    `safe_ratio.sql`'s 4-space indentation inside the `case`, the continuation-line `and`
    alignment, and the trailing blank-line conventions exactly. Do NOT generalize into a
    parameterized macro with a sign-check boolean flag — YAGNI; a five-line sibling macro is
    simpler to read and test (per RESEARCH.md "Don't Hand-Roll").

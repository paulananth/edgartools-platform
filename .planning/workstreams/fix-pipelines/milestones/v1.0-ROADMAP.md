## Milestone Goal

Make pipeline failures impossible to miss — every stage failure surfaces as a hard Step
Functions error, a single status view covers all pipelines completely, and operators get
an SNS email notification when anything fails.

---

## Phases Shipped

### Phase 1: Failure Surfacing ✓
**Goal:** Every stage failure in all 5 state machines propagates to a hard FAILED execution.

**What shipped:**
- Set `ToleratedFailurePercentage: 0` on `BatchBootstrap` (bootstrap_phased) and `BatchSilver` (silver_mdm_gold) Distributed Map states — was 10%
- Increased ECS task `MaxAttempts` from 2 to 3 across all `ecs_state()` calls
- Created `scripts/ops/test-failure-surfacing.sh` — repeatable failure-injection test script
- Added failed-batch recovery runbook section to `docs/runbook.md`

**Plans:** 01-01 (deploy script edits + redeploy), 01-02 (test script + runbook), 01-03 (live failure injection test)

**Key decision:** Recovery path is `targeted_resync` for failed batches; full re-run is also safe (idempotent by DEC-009).

### Phase 3: Failure Notifications ✓
**Goal:** Operators receive SNS email when any pipeline reaches FAILED state.

**What shipped:**
- New Terraform module `infra/terraform/modules/pipeline_notifications/`
  - SNS topic `edgartools-dev-pipeline-failures`
  - SNS topic policy (AllowEventBridgePublish — no Condition per AWS restriction)
  - SNS email subscription → thepaulananth@gmail.com (confirmed)
  - EventBridge rule: single catch-all for all `edgartools-dev-*` Step Functions FAILED events
  - EventBridge target: SNS with input_transformer (human-readable email body, no Lambda)
- Wired into `infra/terraform/accounts/prod/main.tf` with `environment = "dev"` hardcoded
- Deployed to AWS via `terraform apply -target=module.pipeline_notifications`
- Live end-to-end verified: mdm-gold failure → email received within 60 seconds

**Plans:** 03-01 (module + wiring + deploy)

**Key decisions:**
- No Lambda — EventBridge input_transformer only (D-01)
- "Failed stage name" descoped (D-06) — not in EventBridge event schema without Lambda
- Static CloudWatch log group URL (D-07) — no time filter (no arithmetic in input_transformer)
- State machine ARNs use hyphens (`edgartools-dev-bootstrap-phased`), not underscores — research finding

## Milestone Summary

### Shipped Decisions
- D-01: No Lambda for notification formatting — EventBridge input_transformer only
- D-02: Single catch-all EventBridge rule (prefix match on `edgartools-dev-`)
- D-03: New `pipeline_notifications` Terraform module (not inline in warehouse_runtime)
- D-06: "Failed stage name" descoped from OBS-06 — requires Lambda which is forbidden by D-01
- BOOTSTRAP_BATCH_CONCURRENCY default 3 preserved (already within 2–5 safe range)

### Technical Debt / Known Gaps
- OBS-06 partially satisfied: email contains pipeline identity, execution ARN, timestamp, Step Functions console link, and static CWL log group link — but NOT the specific failed stage name (requires Lambda)
- SNS email subject is "AWS Notification Message" (AWS default) — formatted subject appears as first line of body (EventBridge input_transformer cannot set SNS Subject)
- CloudWatch URL has no time filter — operator must manually set time window using stopDate epoch value in email body
- `pipeline_notifications` module deployed via `-target` (prod Terraform root has never been fully applied; other modules represent unmanaged existing infra)
- OBS-01 runtime re-test pending: test-failure-surfacing.sh has a race condition with S3 file overwrite; execution was aborted during Phase 1 validation

### Out of Scope (deferred)
- Slack/PagerDuty notifications
- Cross-task coordinated rate limiting (DynamoDB token bucket)
- Hard validation rejecting BOOTSTRAP_BATCH_CONCURRENCY outside [2, 5] at deploy time
- Batch log correlation improvements (batch-logs.sh CIK-to-task mapping)
- Lambda-based deep-link to specific failing ECS task log stream

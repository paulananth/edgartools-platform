# Phase 7: Relationship Graph Consistency, Temporal Lineage, And Artifact Hygiene - Research

**Researched:** 2026-07-12
**Scope:** PostgreSQL MDM → Snowflake-hosted Neo4j generation publication, temporal relationships, coverage evidence, and relationship-input integrity

## Summary

Phase 7 is an architectural migration, not a documentation-only phase. The current platform
already has useful foundations: date-valued relationship bounds, stable UUID relationship
instances, relationship source mappings, a Snowflake graph contract, identity-level missing/extra
diagnostics, AWS Step Functions/ECS orchestration, and immutable bronze cache checks. The main gap
is publication semantics: graph tables are rebuilt in place, verification is count-led, sync state
is stored per PostgreSQL row, exclusions are implicit constants/prose, and silver publication can
upload a local database without semantic merge or optimistic promotion.

The maintainable path is to evolve these surfaces rather than introduce a second graph runtime:
version MDM-export and graph-contract rows by generation, stage immutable partitions, verify their
manifests and row identities/properties, then switch one Snowflake active-generation pointer used
by both MDM serving and Native App views.

## Existing Architecture Findings

### Relationship model

- `edgar_warehouse/mdm/database.py` defines `MdmRelationshipType`, source mappings, property
  definitions, and `MdmRelationshipInstance`.
- Relationship instances already have a UUID `instance_id`, `Date`-typed `effective_from` and
  `effective_to`, JSON properties, source system/accession, active flag, and audit timestamps.
- `GraphSyncEngine.ensure_relationship()` currently treats bounds, accession, and properties as
  part of duplicate detection. Phase 7 must separate stable logical identity from version identity.
- The existing `MdmSourcePriority` is entity-oriented. Relationship-specific conflict precedence
  needs its own explicit registry or a safely extended rule model.
- Existing migrations are imperative/idempotent in `edgar_warehouse/mdm/migrations.py`; new tables
  and additive columns should follow that pattern.

### Snowflake-hosted Neo4j projection

- `edgar_warehouse/mdm/snowflake_graph.py` is the canonical graph surface.
- `render_graph_tables()` currently runs `CREATE OR REPLACE TABLE MDM_GRAPH_NODES/EDGES AS ...`
  and exposes stable `GRAPH_APP_*`, `GRAPH_NODE_*`, and `GRAPH_EDGE_*` views.
- Neo4j Graph Analytics is hosted inside Snowflake; Aura/Bolt is explicitly retired.
- The current edge contract has `EDGEID`, relationship type, endpoints, source fields,
  `GRAPH_SYNC_STATUS`, `GRAPH_SYNCED_AT`, and JSON properties. Typed validity/provenance and
  `GENERATION_ID` must be added.
- Verification already computes missing/extra edge identities and missing endpoints, so it can be
  extended to property hashes/diffs and exhaustive coverage statuses rather than replaced.
- `POPULATED_RELATIONSHIP_TYPES` is a temporary allowlist for named parity checks. A generation
  coverage manifest should supersede it by requiring every active registered type to declare
  `populated`, `valid_zero`, or `excluded`.

### Export and orchestration

- `mdm export` precedes `mdm sync-graph` in AWS Step Functions definitions generated by
  `infra/scripts/deploy-aws-application.sh`.
- Multiple workflows repeat the linear MDM run → relationship derivation → export → sync → verify
  sequence. The current verifier is sometimes caught and allowed to continue to gold, which is
  incompatible with verified-generation activation.
- AWS deployment already owns ECS task definitions and Step Functions outside Terraform. Phase 7
  should add publisher/coordinator/partition commands and Step Functions Map fan-out here, leaving
  passive Terraform unchanged.
- A transactional PostgreSQL publication request/outbox decouples every writer from graph runtime
  details. Step Functions may poll/claim queued generation work, fan out partitions, fan in
  verification, and activate once.

### Silver and bronze integrity

- `_publish_silver_database_if_remote()` currently uploads the local
  `silver/sec/silver.duckdb` through the storage adapter; existing tests assert direct upload.
- `_localize_silver_database_if_remote()` already provides a natural place to retain source object
  identity and build a merged local candidate.
- `object_storage.py` abstracts local/S3 reads and writes but does not expose a conditional promote
  contract. Phase 7 needs object metadata/version reads, staged writes, and compare-before-promote.
- `bronze_filing_artifacts.py` already skips existing artifact rows/files when `force=False` and
  avoids throttle sleeping on cache hits. `filing_artifact_service.py` is the shared boundary for
  global idempotency tests and repair audit emission.

## Recommended Data Contracts

### PostgreSQL staging/authority

Add explicit logical and temporal version concepts while preserving `instance_id` compatibility:

- Logical relationship ID: immutable across temporal versions and graph generations.
- Relationship version ID: immutable per `[valid_from_date, valid_to_date)` property/provenance version.
- Date provenance enum: `reported`, `filing_date_proxy`, `unknown`.
- Relationship kind: `direct` or governed `derived`.
- Supersession/quarantine fields; no ordinary physical deletion.
- Relationship-specific source-priority/tie-break policy.
- Publication request/outbox row committed transactionally with the MDM watermark.

Prefer additive migration and a compatibility backfill that maps each existing `instance_id` to a
logical relationship/version without changing current IDs in place until all readers migrate.

### Snowflake generation tables

Use immutable generation data plus stable filtered views:

- `MDM_GRAPH_GENERATION`: lifecycle, MDM watermark, content hashes, created/verified/activated audit.
- `MDM_GRAPH_ACTIVE_GENERATION`: one environment-scoped active generation pointer.
- `MDM_GRAPH_PARTITION_MANIFEST`: node/relationship type, shard, fingerprint, hashes, row count,
  retry/reuse state.
- `MDM_GRAPH_COVERAGE`: relationship type, status, expected count, evidence/fingerprint fields.
- Versioned node and edge rows with `GENERATION_ID`; edge rows include stable relationship/version
  IDs, typed dates/provenance, relationship kind, source provenance, and query properties.

Stable views join/filter through the active pointer. One pointer update changes the consumer-visible
node and edge snapshot together. Rollback is the same guarded pointer operation to a retained,
verified generation.

### Parallel build protocol

1. Claim/coalesce queued PostgreSQL publication requests and freeze a committed MDM watermark.
2. Create a generation row in `building` state.
3. Fan out node-type and relationship-type partitions; hash-shard only configured high-volume types.
4. Each worker writes immutable staged rows and a signed/hashed manifest.
5. Reuse a prior partition only when type, shard, input fingerprint, rule/schema version, and MDM
   watermark inputs prove equality.
6. Fan in; verify manifest completeness, node endpoints, identity parity, property parity, temporal
   validity, coverage ledger, and exclusion freshness.
7. Mark verified, atomically switch the shared pointer, and mark publication requests active.
8. On failure, retain the prior pointer, retry failed partitions, and alert by queue age/SLO.

## Temporal Query Semantics

- Business validity uses date-only half-open intervals: start inclusive, end exclusive.
- Null start means unknown beginning; null end means no known ending.
- Typed dates live in both MDM and graph edges; operational timestamps remain audit-only.
- Active graph generations contain all non-quarantined versions, including ended history.
- Current views use today's date. Historical traversal accepts `as_of_date`.
- Strict historical queries exclude an edge unless the date is provably in its interval. An explicit
  `include_unknown_dates` mode may add uncertain edges and must label uncertainty.
- Entity merges remap traversal endpoints to canonical entities while retaining original identifiers
  and merge lineage in provenance.

## Exclusion And Coverage Design

Replace the temporary populated-type allowlist with exhaustive machine-readable coverage:

- `populated`: identity/property parity is proven.
- `valid_zero`: recomputed for this generation with query version, source fingerprint, date, and zero result.
- `excluded`: reviewed evidence is current and expected edge count is zero.

EDGE-07 is `source_unavailable`; its fingerprint includes adviser CIKs/accessions examined for ADV
electronic documents. EDGE-08 is `capability_not_implemented`; its fingerprint includes the company
universe and Exhibit 21 parser-capability version. Any fingerprint change makes the exclusion stale,
which blocks verification and activation. Exclusions never generate fake nodes or edges.

## Semantic Silver Publication

The guard must compare domain semantics, not DuckDB bytes:

- A reviewed table-policy registry declares business keys, domain/ephemeral classification,
  authority/conflict rules, and compatible schema policy.
- Local partial databases merge into a localized canonical database.
- Protected canonical keys cannot disappear through ordinary publish.
- Same-key differing rows resolve only through declared provenance rules; ambiguity aborts with a diff.
- Schema changes are additive-only in ordinary publication.
- Upload to a staged S3 key, validate checksum/database health, and promote only when the source
  canonical ETag/version remains unchanged. Conflict is retryable, not last-writer-wins.
- Destructive corrections use a separate dry-run/audited repair command; silver `--force` cannot
  disable the guard.

## Risks And Mitigations

- **Scope growth:** Eight plans isolate live Native App feasibility, schema/temporal semantics,
  coverage, queueing, parallel assembly, activation, artifact integrity, and live rehearsal.
- **Distributed atomicity:** Do not claim atomic PostgreSQL+Snowflake commits. Consumer consistency
  comes from the shared Snowflake active pointer; PostgreSQL is staging/derivation authority.
- **Native App compatibility:** Preserve stable existing view names and additive columns; validate
  current Neo4j procedure expectations before activation.
- **Historical volume:** Type-first partitions, selective hash sharding, and content-addressed reuse
  control rebuild cost. Full history remains logically present in each active generation.
- **False partition reuse:** Include source watermark, rule version, schema version, and input hashes
  in reuse identity; fail closed on absent fields.
- **Operational lag:** Queue lifecycle metrics, five-minute warning, fifteen-minute hard alert, and
  bounded backfill windows distinguish consistency from freshness.
- **Migration safety:** Backfill IDs/dates additively, retain compatibility views, and prove rollback
  before removing mutable rebuild behavior in a later phase.

## Validation Architecture

### Fast feedback

- Unit tests for interval semantics, stable IDs/versioning, source-priority conflicts, coverage
  fingerprints, partition fingerprints/reuse, and protected-table merge policy.
- SQL rendering tests for generation columns, active-pointer filtering, typed temporal fields, and
  exhaustive identity/property parity queries.
- Step Functions definition tests proving distributed Map fan-out, failure fan-in, no passive
  Terraform workload ownership, and activation only after verification.

### Integration

- PostgreSQL migration/outbox tests use the repository's existing MDM database fixtures.
- Fake/temporary Snowflake connector tests execute rendered statements and assert pointer behavior.
- DuckDB/local object-store fixtures prove semantic merge, ambiguity abort, and schema policy.
- Artifact service tests prove zero download callback calls on intact cache hits and complete repair audit.

### Live dev rehearsal

Use `SNOW_CONNECTION=snowconn` for dev DDL/queries. Build a bounded generation, validate all coverage
statuses, run current and historical multi-hop queries, inject one partition failure, retry/reuse,
inject a stale exclusion, simulate concurrent silver promotion, activate, and roll back. Capture
generation IDs, hashes, counts, parity diffs, query results, alerts, and rollback evidence in
`07-VERIFICATION.md` or a dedicated rehearsal evidence document.

<objective>
Add named per-type parity assertions to `mdm verify-graph` for the 6 node types and the 4
already-populated relationship types (NODE-01..06, EDGE-01..04), per CONTEXT D-01.

`verify()` in `edgar_warehouse/mdm/snowflake_graph.py` already computes per-type parity rows
(`node_parity["by_entity_type"]`, `relationship_parity["by_relationship_type"]` via
`_render_verify_node_counts` / `_render_verify_relationship_counts`), but its pass/fail gate
only checks the *aggregate* `node_parity["status"]` / `relationship_parity["status"]`. Those
aggregate statuses are computed from a `FULL OUTER JOIN` — so an expected type that is entirely
absent from both sides (e.g. `audit_firm` before its view exists, or a populated edge type that
silently dropped to zero on both sides) contributes no row and keeps the aggregate `ok`. That is
the silent-omission gap NODE-01..06 / EDGE-01..04 must close.

This plan adds a `named_checks` list to the verify payload: one named check per expected node
type and one per populated relationship type. Each check asserts the type is *present* in the
parity rows AND at parity (mdm_minus_graph == 0 and graph_minus_mdm == 0). A missing named type
is a hard failure. The aggregate of these named checks is folded into `verify()`'s `passed` gate
so the CLI exit code and Step Functions `mdm_verify_graph` gate now catch a missing per-type
view. Per D-01 these are named assertions over data `verify()` already returns — no new SQL,
no new command, keeping `verify-graph` the single atomic gate `go-live.sh` and Step Functions
already depend on.

Purpose: NODE-01..06 and EDGE-01..04 require a *named, per-type* parity assertion for each of
the 6 node types and 4 populated edge types, not just an aggregate pass. NODE-06 (audit_firm)
becomes checkable now that plan 05-01 emits the `GRAPH_NODE_AUDITFIRM` view.
Output: a `named_checks` payload section in `verify-graph` covering the 6 node + 4 edge types,
wired into the exit-code gate, with CLI test coverage for the present, mismatched, and
missing-type cases.
</objective>

| Threat ID | Category | Component | Severity | Disposition | Mitigation Plan |

| T-05-04 | Spoofing / false-negative | verify()'s named-check gate treating an absent expected type as ok | medium | mitigate | Named checks default `present=False`→`status=failed` for any expected type missing from the parity rows, and their aggregate is folded into the passed gate — an omitted type now fails verification instead of passing silently. This is the exact silent-omission gap NODE/EDGE checks close. |
| T-05-05 | Denial of service / false-positive | named-checking the 7 not-yet-populated relationship types would make verify-graph fail on legitimately-zero data this milestone does not cover | low | mitigate | POPULATED_RELATIONSHIP_TYPES scopes named edge checks to exactly the 4 EDGE-01..04 types; the 7 zero types are excluded with a documented comment until Phases 6-7 populate them. |
| T-05-06 | Tampering | named-check helpers read parity data only, add no SQL | low | accept | Helpers consume the already-computed node_parity/relationship_parity payload; no new query string reaches Snowflake, so the identifier-allowlist surface is unchanged. |
</threat_model>

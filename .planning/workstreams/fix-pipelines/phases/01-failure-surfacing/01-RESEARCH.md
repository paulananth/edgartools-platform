## Summary

Phase 1 has two edit targets in a single file (`infra/scripts/deploy-aws-application.sh`) and two new deliverables (`scripts/ops/test-failure-surfacing.sh`, a new runbook section appended to `docs/runbook.md`). All locked decisions are implementable without ambiguity.

The failure-injection test mechanism is resolved: `bootstrap_phased` does not accept a CIK list via execution input — it calls `seed-universe` which writes `cik_batches.jsonl` to S3. The test script must overwrite this file with `{"cik_list": "9999999"}` after `SeedUniverse` exits but before `BatchBootstrap` reads it (Option A). CIK `9999999` is confirmed to cause a hard ECS task failure (non-zero exit): `_download_sec_bytes` raises `WarehouseRuntimeError` for non-retried HTTP status codes including 404, which propagates up unhandled and terminates the container non-zero.

The Step Functions behavior at `ToleratedFailurePercentage: 0` is verified against AWS official documentation: the default when the field is omitted is already 0, so explicit `0` and omission are semantically identical. The error propagated to the parent execution is `States.ExceedToleratedFailureThreshold`.

**Primary recommendation:** Plan the code edits as four targeted literal changes in two Python heredocs. The test script task is the most complex deliverable — plan it with the S3 overwrite sequencing strategy (Option A) and a 45-minute timeout budget.

### Anti-Patterns to Avoid

- **Editing only one heredoc:** `ecs_state()` is defined in two independent Python subprocess blocks. Updating only `write_bootstrap_phased_definition` leaves `write_silver_mdm_gold_definition` at `MaxAttempts: 2`.
- **Using a short timeout in the test script:** `SeedUniverse` calls SEC EDGAR (5-10 min). Then with `MaxAttempts: 3`, `IntervalSeconds: 120`, `BackoffRate: 2.0`, the child ECS task can take up to 17 min of retry backoff alone. Budget 45 minutes total, not 30.
- **Pre-writing S3 before `SeedUniverse` runs:** `SeedUniverse` overwrites `cik_batches.jsonl` — any pre-write is lost. The injection must happen after `SeedUniverse` exits.
- **Assuming `targeted_resync` takes a CIK list:** It does not. It takes `scope_type` + `scope_key` for a single scope. For batch recovery, the recommended path is a full `bootstrap_phased` re-run (DEC-009 idempotency skips already-loaded CIKs).

## ToleratedFailurePercentage: 0 — Verified Behavior

[VERIFIED: AWS official documentation — https://docs.aws.amazon.com/step-functions/latest/dg/state-map-distributed.html]

**Default value:** The default value when `ToleratedFailurePercentage` is omitted is 0. Setting it explicitly to `0` is semantically identical to omitting it. The change from `10` to `0` removes the 10% tolerance window.

**Error propagated to parent:** `States.ExceedToleratedFailureThreshold`

This error name appears in:
- The parent execution's failure event (`executionFailedEventDetails.error`)
- The Map Run's failure details accessible via `describe-map-run`

**Important AWS caveat (from official docs):** "Step Functions may continue to run child workflows in a Map Run even after the tolerated failure threshold is exceeded, but before the Map Run fails." This means in a 10-batch run, 1-2 additional batches may complete after the first failure before the Map Run stops. This is by-design behavior — the parent execution still reaches FAILED state decisively.

**Percentage calculation:** failed_items / total_items. With a single-item test (`cik_batches.jsonl` with one line), 1/1 = 100%, which exceeds 0%. Threshold trigger is guaranteed for a single-item test.

**Explicit 0 vs. omission:** AWS docs confirm both produce identical behavior. The codebase currently has `"ToleratedFailurePercentage": 10` which overrides the default — changing to `0` is the minimal correct edit (as opposed to removing the key entirely, which preserves readability for future engineers).

## Failure-Injection Test: CIK Injection Mechanism (RESOLVED)

[VERIFIED: warehouse_orchestrator.py lines 843-906, 1016-1070, 1368-1411, 1692-1731; sec_client.py lines 42-96; dataset_path_catalog.py line 144]

### How bootstrap_phased acquires its CIK list

1. `SeedUniverse` calls `edgar-warehouse seed-universe --run-id <execName>`
2. `seed-universe` fetches SEC EDGAR company_tickers.json, deduplicates, filters against MDM/silver active CIKs, then calls `_write_cik_universe_batches()` which writes to S3 at:
   ```
   warehouse/bronze/reference/cik_universe/runs/<execName>/cik_batches.jsonl
   ```
   Each line: `{"cik_list": "cik1,cik2,..."}` (100 CIKs per line, batch_size=100)
3. `BatchBootstrap` Map state's `ItemReader` reads from that exact S3 key (line 1355 of deploy-aws-application.sh)

### Does CIK 9999999 cause a non-zero ECS exit? YES — VERIFIED

[VERIFIED: sec_client.py lines 42-96]

`_download_sec_bytes()` in `sec_client.py`:
- Line 77: only retries on HTTP status codes `{429, 500, 502, 503, 504}`
- Line 96: for all other status codes (including 404): `raise WarehouseRuntimeError(f"SEC request failed for {url}: HTTP {status_code}")`
- This exception is NOT caught anywhere in the `bootstrap-batch` call chain: `_capture_submissions_main` → `_capture_submission_bronze_snapshot` → `_run_submissions_bronze_then_silver` → bootstrap-batch handler → `_execute_warehouse` → ECS task exits non-zero

CIK `9999999` does not exist in SEC EDGAR. A 404 response raises `WarehouseRuntimeError` immediately (no retries at the SEC client level). The ECS container exits non-zero. Step Functions records this as `States.TaskFailed`. After 3 SFN-level retry attempts (with 120s, 240s, 480s backoff), the child execution reaches FAILED state.

### Injection strategy: Option A (S3 overwrite after SeedUniverse)

The test script:
1. Derives the execution name deterministically: `test-failure-surfacing-$(date -u +%Y%m%d-%H%M%S)`
2. Starts `bootstrap_phased` with that name
3. Polls `get-execution-history` every 15s until `stateExitedEventDetails.name == "SeedUniverse"` appears
4. Overwrites `cik_batches.jsonl` on S3 with a single line: `{"cik_list": "9999999"}`
5. Polls `describe-execution` every 30s until status is not RUNNING (timeout: 45 minutes)
6. Asserts status == FAILED; exits 1 if SUCCEEDED or timed out

The sequential state machine guarantees that `BatchBootstrap`'s `ItemReader` cannot start before `SeedUniverse` exits — so there is no race condition between the SFN state transition and the S3 overwrite, provided the overwrite completes before the Map state's `ItemReader` call. In practice, Step Functions has a brief (~1-5s) internal scheduling lag after a state exits before the next state's resource calls execute.

**BRONZE_BUCKET_NAME** is available as an env var in `deploy-aws-application.sh` and must be passed to the test script (or read from `$BRONZE_BUCKET` env var, matching existing ops script conventions).

### Why Option B (silver_mdm_gold) does not work as test vehicle

`silver_mdm_gold` calls `seed-silver-batches` which reads from silver DuckDB. CIK `9999999` does not exist in silver. The resulting `cik_batches.jsonl` would contain zero lines (empty batch). A Distributed Map with zero items succeeds with zero failures — the execution reaches SUCCEEDED, not FAILED. This does not test the failure-surfacing behavior.

## Common Pitfalls

### Pitfall 1: Editing only one heredoc
**What goes wrong:** `write_bootstrap_phased_definition` is updated but `write_silver_mdm_gold_definition` still has `MaxAttempts: 2` and `ToleratedFailurePercentage: 10`. OBS-02 (silver_mdm_gold) is not satisfied.
**Why it happens:** `ecs_state()` looks like a shared function but is defined twice, once per heredoc subprocess.
**How to avoid:** Use the line-range grep after editing: `sed -n '1288,1530p' infra/scripts/deploy-aws-application.sh | grep '"MaxAttempts": 2'` — must return no output.
**Warning signs:** Grep still shows occurrences after editing.

### Pitfall 2: Test script timeout budget insufficient
**What goes wrong:** Test script polls for 30 minutes; execution is still RUNNING; script times out.
**Why it happens:** `SeedUniverse` calls SEC EDGAR (~5-10 min). After injection, the child ECS task with `MaxAttempts: 3` and `IntervalSeconds: 120`, `BackoffRate: 2.0` takes: 120s + 240s + 480s = 840s retry backoff + ECS startup (~45s each × 4 attempts = 180s) + task execution time = ~1020s (~17 min) for the child to reach FAILED. Total wall time from execution start: SeedUniverse (5-10 min) + injection (near-instant) + child failure time (17 min) = 22-27 min + SFN overhead.
**How to avoid:** Set `MAX_WAIT_SECONDS=2700` (45 minutes) in the test script. Use 30-60s sleep between status polls.
**Warning signs:** Script exits with "timed out" rather than FAILED assertion.

### Pitfall 3: Pre-writing S3 before SeedUniverse completes
**What goes wrong:** Test script writes `cik_batches.jsonl` before starting the execution. `SeedUniverse` ECS task overwrites the file with real CIK data. BatchBootstrap processes real CIKs and may succeed.
**Why it happens:** The execution name is known before `start-execution`, but `seed-universe` always writes to S3 when it runs.
**How to avoid:** Poll `get-execution-history` for the `stateExitedEventDetails.name == "SeedUniverse"` event before writing to S3.
**Warning signs:** Execution reaches SUCCEEDED or runs much longer than expected (processing the full real CIK universe).

### Pitfall 4: Definition file vs. deployed state machine mismatch
**What goes wrong:** Verification checks the locally generated JSON temp file and finds `ToleratedFailurePercentage: 0`, but the deployed state machine was never updated (deploy script failed partway through).
**Why it happens:** `write_*_definition()` writes a temp file; `upsert_state_machine()` reads and deploys it — both must succeed.
**How to avoid:** Verification task must also query the deployed SM: `aws stepfunctions describe-state-machine --state-machine-arn <arn> --query 'definition' | grep ToleratedFailurePercentage`.
**Warning signs:** Definition file looks correct but `test-failure-surfacing.sh` shows execution reaching SUCCEEDED.

### Pitfall 5: Post-failure child executions still running
**What goes wrong:** Operator sees the Map Run as FAILED but notices some child executions started after the first failure also ran. Incorrectly concludes `ToleratedFailurePercentage: 0` is not working.
**Why it happens:** AWS explicitly documents: "Step Functions may continue to run child workflows in a Map Run even after the tolerated failure threshold is exceeded, but before the Map Run fails." This is by-design.
**How to avoid:** Document in runbook. Verify behavior by checking parent execution status (FAILED), not individual child execution counts.

## Open Questions

1. **A3: S3 overwrite race window in Option A**
   - What we know: SFN sequential states transition within 1-5s; `aws s3 cp` completes in ~1s from a same-region EC2/Cloud9 or ~2-3s from developer laptop.
   - What's unclear: Whether the SFN execution history event `stateExitedEventDetails` appears before or after the internal state transition to `BatchBootstrap` begins. If the history event is written concurrently with the next state starting, there is a small race.
   - Recommendation: Add a `sleep 3` after detecting the SeedUniverse exit event, then overwrite S3. This is a trivially cheap delay that eliminates the race in practice.

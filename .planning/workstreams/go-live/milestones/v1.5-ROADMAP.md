# Milestone v1.5: Go Live

**Status:** ✅ SHIPPED (planning/evidence complete) 2026-06-19
**Phases:** 1-5
**Total Plans:** 12

## Overview

Prepare the AWS-first EdgarTools Platform for production go-live by proving the launch
gates for AWS deployment, Snowflake native pull and gold, MDM Snowflake Postgres,
Snowflake-hosted graph verification, dashboard inspection, secret-safe evidence, and
operator handoff.

This milestone is launch-readiness evidence and operator handoff, not a production
deploy. The final deliverable — `05-GO-NO-GO-PACKET.md` — records the actual launch
decision as **NO-GO — Conditional**: 7 of 16 requirements (LIVE-03, MDM-01, GRAPH-01,
GRAPH-02, DASH-01, DASH-02, DASH-03) require real production credentials/infrastructure
that do not yet exist, and remain documented as launch blockers with remediation steps
rather than completed work. This is the intended, by-design outcome of the milestone —
dev rehearsal proven end-to-end, prod proof deliberately deferred to an explicit future
production-launch action gated on operator approval.

## Milestone Summary

**Key Decisions:**

- Treat go-live as v1.5 and keep phase numbering local to this isolated workstream (`.planning/workstreams/go-live/`).
- Keep AWS as the only active deployment path; Terraform remains passive infrastructure only.
- Use existing deploy and verification scripts before adding automation — no new non-AWS architecture.
- `edgar-warehouse mdm verify-graph` is the hosted graph acceptance gate (strict mode).
- Dashboard launch evidence is operator inspection evidence; it does not replace CLI acceptance.
- No secrets, DSNs, tokens, raw connector errors, Terraform state, or sensitive generated deployment values may be committed anywhere in the evidence bundle — secret NAMES are allowed, values never are.
- The go/no-go packet synthesizes the launch gate matrix into 5 blocker themes rather than duplicating individual gate rows (D-01); it links to the matrix as the single source of truth.
- Post-launch monitoring checklist documents exactly the 8 OPS-02 systems with read-only diagnostics only.
- Follow-up production work is captured in repo `TODOS.md` (append-only, What/Why/Where format) rather than expanding the launch milestone.

**Issues Resolved:**

- Dashboard README's lingering external Neo4j (`NEO4J_*`) references removed; architecture test gate now enforces their absence.
- `bootstrap-batch` removed from `GOLD_AFFECTING_COMMANDS` (separately documented in `CLAUDE.md`'s long-load 5-whys) — unblocked the phased pipeline this milestone's evidence relies on.

**Issues Deferred (Known Gaps — by design, documented in `05-GO-NO-GO-PACKET.md`):**

These 7 requirements are intentionally left unchecked because they require real production
credentials/infrastructure that this milestone's scope (dev rehearsal + evidence capture)
does not provision. They are the milestone's actual deliverable — a precise, named list of
what blocks GO — not missed work:

- **LIVE-03 / MDM-01 / GRAPH-01 / GRAPH-02** (Phase 3): production MDM Secrets Manager values not yet populated; no production-target `mdm sync-graph`/`mdm verify-graph`/AWS MDM E2E run yet.
- **DASH-01 / DASH-02 / DASH-03** (Phase 4): no UAT pass yet recorded against a production or production-like read-only dashboard configuration.

Each gap has a named remediation step and owner in `05-GO-NO-GO-PACKET.md`'s 5 NO-GO
Blockers section and `runbook/launch-ops.md` / `runbook/post-launch-monitoring.md`.

**Technical Debt Incurred:**

- `EDGARTOOLS_DEV_DEPLOYER` lacks direct `SELECT` grant on `EDGARTOOLS_SOURCE`, blocking `dbt run --full-refresh` in dev (tracked in `CLAUDE.md` and `TODOS.md`); the analogous `EDGARTOOLS_PROD_DEPLOYER` grant must be confirmed before any production `dbt run --target prod --full-refresh`.
- External Neo4j runtime remnants beyond the dashboard README (already cleaned in Phase 4) remain for future formal deprecation.

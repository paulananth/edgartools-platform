## UPDATE (this session, later): Option B bronze sync done, but won't unblock silver
Bronze sync (dev → prod, `aws s3 sync`) completed successfully: prod bronze now
has exactly 145,908 objects / 3,777,512,484 bytes, matching dev exactly
(`SYNC_EXIT=0` in `/tmp/bronze-sync.log`).

**However, starting `edgartools-prod-silver-mdm-gold` would NOT build silver
from this copied bronze.** Traced the actual code
(`edgar_warehouse/application/warehouse_orchestrator.py:1326-1345`):
`seed-silver-batches` (the 2nd state in that pipeline, "SeedSilverBatches")
calls `db.get_ciks_with_bronze(...)`, which queries silver DuckDB's
`sec_company_sync_state` JOIN `sec_source_checkpoint` (source_name=
'submissions_main') tables — **not** anything in S3/bronze. On a fresh prod
silver DuckDB (auto-created empty by `SilverStore.__init__`,
`silver_store.py:585-589` — does NOT error on missing file), those tables are
empty, so the query returns 0 rows. The orchestrator handles this explicitly
(line 1337-1345): emits `seed_silver_batches_completed` with `cik_count=0,
batch_count=0` and **returns early — no error, no batch file written**.
Net effect: the whole Step Function would "succeed" while doing nothing.

Root cause: `sec_company_sync_state`/`sec_source_checkpoint` rows are written
as a *side effect* of a CIK's bronze+silver having been processed through the
normal bootstrap path at least once — they are not derivable from bronze
object existence in S3. Copying bronze files into prod's bucket does not
backfill this silver-side bookkeeping. This pipeline
(`silver_mdm_gold`/`SeedSilverBatches`) is designed to *reprocess* CIKs silver
already knows about from cached bronze — it is not a bootstrap-from-bronze
mechanism for a brand-new environment.

**Decision needed from user before proceeding**: Option B as approved does
not achieve "prod builds silver from copied bronze with zero SEC calls" via
this state machine. Did not start the execution — would have been a silent
no-op, wasting an opaque run and creating false-positive evidence. Options to
discuss next session: (a) run a real `load_history`/`bootstrap` flow scoped to
the actual prod target universe (incurs real SEC calls, contradicts the
"reuse dev's already-fetched data" goal), (b) find/write a one-off command
that backfills `sec_company_sync_state`/`sec_source_checkpoint` directly from
bronze object listing (no such command currently exists — would need new
code), or (c) revisit Option A (copy dev's already-built `silver.duckdb`
directly) despite its earlier-flagged downside (reflects dev's test universe,
not necessarily prod's intended universe).

## 5-whys for the bucket-naming bug (bug #4)
1. **Symptom**: `mdm_run` ECS task exits 1 with `MDM_SILVER_DUCKDB` bucket-not-found.
2. **Why**: task def env var pointed at `edgartools-prod-warehouse-077127448006`.
3. **Why**: `deploy-aws-application.sh` defaults bucket names with
   `${NAME_PREFIX}-<purpose>-${ACCOUNT_ID}` unconditionally.
4. **Why**: that convention is correct for dev (Terraform appends account id
   there) but prod's Terraform root does not append it — the script assumed
   one fixed naming convention across environments instead of asking AWS
   or Terraform what the real name is.
5. **Why it recurred across multiple deploys**: the first wrong value got
   written into `infra/aws-prod-application.json`, and the script's
   parameter-resolution order prefers the cached manifest value over
   re-deriving it, so the bug self-perpetuated until someone passed an
   explicit override (as done this session) or fixed the default logic.
   **Root-cause fix applied**: derive the bucket name from what actually
   exists in S3 (`resolve_bucket_name()`), not from a hardcoded convention.

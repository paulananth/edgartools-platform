## NEW finding this session: third bug, different class entirely
E2E re-run #3 (`run-aws-mdm-e2e.sh --env prod ...`) got PAST both secret
failures — the MDM container started, connected to Postgres via
`MDM_DATABASE_URL`, and successfully ran 13+ `CREATE TABLE IF NOT EXISTS`
statements from `001_initial_schema.sql`. It then failed on:

```
psycopg2.errors.InsufficientPrivilege: must be owner of table mdm_entity_attribute_stage
[SQL: CREATE INDEX IF NOT EXISTS idx_attr_stage_entity_field ON mdm_entity_attribute_stage(entity_id, field_name)]
```//cause JSON had ECS task arn:aws:ecs:...task/edgartools-prod-warehouse/633806ee68054c3a958895621d8bd0f5, ExitCode 1, StopCode EssentialContainerExited.

**This is NOT a secret-wiring bug.** It's a genuine Postgres role/ownership
mismatch: the role used to connect via `MDM_DATABASE_URL` does not own
table `mdm_entity_attribute_stage`, so `CREATE INDEX` (which requires table
ownership in Postgres, unlike `CREATE TABLE IF NOT EXISTS` which succeeded
because the table didn't exist yet or was already owned) fails.

Hypothesis: `mdm_entity_attribute_stage` (and possibly other
`mdm_*` tables) were created by a DIFFERENT Postgres role in an earlier
session/attempt (e.g. an admin/owner role used during manual debugging,
or by `infra/snowflake/sql/postgres/mdm_post_restore.sql` running under
a different connection), and the role embedded in the current
`MDM_DATABASE_URL` secret is not that owner and lacks `ALTER`/superuser
grants on it.

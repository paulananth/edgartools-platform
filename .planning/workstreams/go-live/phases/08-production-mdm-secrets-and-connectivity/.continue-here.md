## Critical Anti-Patterns

| Pattern | Description | Severity | Prevention Mechanism |
|---------|-------------|----------|---------------------|
| Assuming "never exposed in chat" means "safely recoverable" | The 3rd rotation's credential was correctly redacted from the transcript, but that's not the same as being *stored* anywhere — `execute_stage()` discards raw output after the redaction pipe, it never persists it | blocking | Treat any credential-emitting command's output as gone forever the moment the tool call completes, unless you explicitly captured it to a secret store in the same pipeline |
| Splitting a rotation command and its consumer into separate tool calls | Two prior leak incidents this workstream both came from a credential surfacing in an intermediate tool result, not the final one | blocking | Always pipe credential-emitting commands directly into their consumer (e.g. `--dsn-stdin`, `put-secret-value --secret-string file:///dev/stdin`) within one Bash call |
| Re-running an already-completed deployment stage "just to be safe" | Re-ran `deploy-snowflake-stack.sh --run-validation` for an already-PASS Blocker 3 this session, hit an avoidable Terraform-provider credential error | advisory | Check evidence files for a final PASS disposition before re-running any state-changing stage |
| Trusting STATE.md blocker text without checking phase evidence | STATE.md said Blocker 3 was still open; Phase 07's own evidence files said PASS | advisory | Cross-check `evidence/*.md` final-disposition sections, not just the top-level STATE.md blocker list |

**Rule going forward:** any command that might emit a credential must be piped directly into its
final, non-printing consumer in the same tool call. Never let a credential cross a tool-call
boundary unredacted.

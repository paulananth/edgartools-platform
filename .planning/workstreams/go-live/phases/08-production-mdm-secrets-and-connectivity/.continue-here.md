Structural mitigation: NEVER `cat`/`grep` a credentials file without piping through a value-redacting filter first (e.g. `grep -v -i "password\|secret\|token"` on every line of output, or `sed`-mask the value, BEFORE the first read, not after). Treat that password as already compromised — recommend rotation to the user; do not wait for them to ask.

**Do not proceed until the box above is checked (i.e., you understand and will apply the redaction-first rule for any future credentials-file reads).**

## Critical Anti-Patterns

| Pattern | Description | Severity | Prevention Mechanism |
|---------|-------------|----------|---------------------|
| Secret printed via cat/grep | `cat connections.toml \| grep -A6` showed the literal password in the tool result, visible to the user and persisted in conversation history | blocking | Always pipe credential-file reads through `grep -v -i "password\|secret\|token\|dsn"` or read only specific non-secret keys (e.g. `grep -E "^(account|user|database|role|warehouse) ="`) — never a raw `cat` or unfiltered `grep -A` on a file known to contain credentials |
| Bare account locator for SnowCLI connection | `account = "YW12138"` alone 404'd; Snowflake CLI needs `org-account` format (`IXKYQWK-YW12138`) or `account.region.cloud` format, even though Terraform's `account_name`/`organization_name` split works fine for the provider | advisory | When adding a new `snow connection`, always combine `organization_name-account_name` into one identifier, not just `account_name` |
| Assuming prod=dev Snowflake account | User initially asked to "use snowconn to create the edgartools-prod connection" — could have been misread as pointing prod at the dev account | advisory | Always check `infra/terraform/snowflake/accounts/prod/terraform.tfvars` (if present, gitignored) for the real account locator before creating any prod-labeled connection; confirmed here that prod (`YW12138`/org `IXKYQWK`) is a genuinely separate account from dev (`np05015.ca-central-1.aws`) |

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| operator CLI -> Snowflake/dbt | dbt run creates or replaces production gold objects and consumes warehouse credits. |
| dbt adapter output -> committed evidence | dbt logs can contain account identifiers, compiled SQL, traces, or connector errors. |
| production grants -> dbt execution | Missing direct grants can cause dynamic-table refresh failures even when admin queries work. |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-07-06 | Information Disclosure | dbt logs, profiles, and Snowflake query output | mitigate | Commit sanitized summaries only; never stage `profiles.yml`, compiled SQL, raw adapter traces, account locators, passwords, tokens, or raw row dumps. |
| T-07-07 | Tampering | production gold dynamic tables | mitigate | Explicit approval gate before dbt run; dbt commands are exact and run only after native-pull PASS and grant preflight. |
| T-07-08 | Elevation | deployer role grants | mitigate | Preflight direct grants before dbt; missing grants stop early and route remediation to Snowflake access Terraform rather than ad hoc broad grants. |
| T-07-09 | Repudiation | treating partial dbt success as launch pass | mitigate | `dbt run` plus `dbt test` plus freshness/status audit required before SNOW-04 PASS; failed tests keep row BLOCKED. |
| T-07-10 | Denial of Service | expensive or repeated production dbt runs | mitigate | Run the standalone command sequence once after approval; failures produce evidence and stop rather than looping. |
| T-07-SC | Tampering | transient `dbt-snowflake` dependency resolution | mitigate | Use repo-standard `uv run --with dbt-snowflake`; no dependency files are edited or committed by this plan. |
</threat_model>

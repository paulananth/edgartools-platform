<objective>
Implement `edgar-warehouse parse-adv-bronze` as a bounded, idempotent operator command that parses
already-captured ADV bronze artifacts into the current silver ADV tables.

This phase wires the CLI, command registry, orchestrator branch, path catalog, and focused tests. It
must not fetch ADV artifacts from SEC, list S3 prefixes, run gold/dbt/Snowflake exports, or require
serving export/MDM settings. Missing alternate SEC URL artifacts remain a separate backlog item
captured as Phase 999.1.
</objective>

<threat_model>
  <threat id="T-09-01" severity="high" surface="network side effects">
    The ADV parse command could accidentally call SEC fetch/refresh code for missing artifacts.
    Mitigation: use only Phase 8 discovery/read helpers and `object_storage.read_bytes`; tests patch
    `download_sec_bytes` and `refresh_filing_artifacts` to raise if called.
  </threat>
  <threat id="T-09-02" severity="medium" surface="operator artifact input">
    Explicit artifact inputs could bypass storage policy or become dynamic SQL identifiers.
    Mitigation: parse `--artifact` into fixed fields, let Phase 8 ADV form allowlist validate form
    values, pass storage paths only to `read_bytes`, and keep SQL table names fixed.
  </threat>
  <threat id="T-09-03" severity="medium" surface="idempotency and row duplication">
    Reprocessing could duplicate ADV detail rows or waste reads on already parsed accessions.
    Mitigation: skip accessions already present in `sec_adv_filing` before storage read and still
    rely on existing `merge_adv_*` upserts as a second idempotency layer.
  </threat>
  <threat id="T-09-04" severity="medium" surface="runtime export requirements">
    Adding the command to gold/export command sets could require `SERVING_EXPORT_ROOT` or trigger
    unrelated Snowflake/gold behavior.
    Mitigation: keep `parse-adv-bronze` out of `GOLD_AFFECTING_COMMANDS`,
    `SNOWFLAKE_EXPORT_COMMANDS`, and `SERVING_EXPORT_COMMANDS`; add tests or source checks.
  </threat>
</threat_model>

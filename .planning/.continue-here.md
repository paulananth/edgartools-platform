5. **DuckDB migration for existing tables**: `CREATE TABLE IF NOT EXISTS`

   carrying it forward is pointless. Recommended: in `_DDL`/init path, detect
   old-PK table (e.g. via `PRAGMA table_info` or a schema-version marker
   table) and `DROP TABLE` + recreate with new PK — then re-populate via
   `bootstrap-fundamentals --mode entity-facts --force` per CIK (idempotent,
   re-fetchable per CLAUDE.md SEC idempotency rules). Confirm with user before
   adding any DROP TABLE migration logic — needs to not nuke other
   tables/namespaces, and needs to be safe to run against the shared
   fundamentals shards.

### Stage 2 — period_start for residual 2,440 collisions (QTD vs YTD duration facts)

7. **`edgar_warehouse/parsers/financials.py`**: add
   `"period_start": fact.get("start")` to `_extract_financial_fact_row`'s
   returned dict. New nullable `period_start DATE` column on
   `sec_financial_fact`.

8. **PK with nullable column problem**: SQL PKs require NOT NULL columns.
   Instant-fact rows have `start=None`. Options (needs design decision,
   flagged for discussion, not decided):
   - (a) Use a sentinel date (e.g. `'0001-01-01'`) for NULL `period_start` in
     the PK, store real NULL in a separate display column, OR
   - (b) Make `period_start` NOT NULL with the sentinel as the actual stored
     value (simpler, slightly less clean semantically).
   Recommend (b) for simplicity unless a consumer needs to distinguish
   "instant" vs "duration with unknown start".

9. **`sec_financial_derived` Stage 2 implications**: `compute_derived_for_accession`
   is called once per `(accession_number, fiscal_period, fiscal_year,
   period_end, form_type)` group (`fundamentals_ingest.py:200-208`) using
   ALL fact rows matching that group — if a concept has both QTD and YTD
   variants in that group (same period_end, different period_start), the
   concept-priority lookup in `financials_derived.py` may pick whichever
   appears first, which could be wrong (e.g. picking YTD revenue for a
   "quarterly" derived row). This needs explicit handling — likely filter
   `fact_rows` to the QTD variant (period_start closest to period_end, i.e.
   shortest duration) before passing to `compute_derived_for_accession`.
   FLAG FOR DESIGN DISCUSSION before implementing Stage 2.
